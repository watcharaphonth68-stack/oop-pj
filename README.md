# 2. Project Description
โปรเจกต์เกม Tetris คลาสสิกที่เขียนและพัฒนาด้วยภาษา Python โดยมีการใช้ไลบรารี Pygame เป็นแกนหลักในการเรนเดอร์กราฟิกตัวเกมและจัดการการรับค่าจากคีย์บอร์ด

# 3. Features
- **Block falling**: บล็อกตกลงมาตามแรงโน้มถ่วง พร้อมความเร็วที่เพิ่มขึ้นตาม Level ของผู้เล่น
- **Move left/right**: เลื่อนบล็อกไปซ้ายและขวาได้อย่างอิสระและลื่นไหล
- **Rotate block**: รองรับการหมุนรูปทรงบล็อกเพื่อจัดวาง
- **Line clear**: ล้างแถวเมื่อบล็อกเรียงเต็มแถบแนวนอน
- **Score system**: ระบบนับคะแนนและระบบเลเวลที่ท้าทายมากยิ่งขึ้น
- **Hard Drop**: วางบล็อกลงพื้นพริบตาเดียวแบบรวดเร็ว
- **Next / Hold Piece**: ดูชิ้นส่วนบล็อกใบหน้าถัดไป และระบบเก็บชิ้นส่วนไว้ใช้งานยามฉุกเฉิน
- **Game over & Restart**: ระบบจบเกมเมื่อบล็อกซ้อนกันถึงบนสุด พร้อมปุ่มกดเริ่มเกมตากระดานใหม่

# 6. Project Structure
โครงสร้างโฟลเดอร์ของตัวเกมถูกแบ่งตามหลักการเพื่อความง่ายในการ Maintain:

- `core/`: จัดเก็บไฟล์ควบคุมลูปกติกาหลักๆ เช่น ไฟลจัดการหน้าต่างทำงาน `game.py`, ระบบกระดาน Grid `board.py`, ข้อมูลตั้งค่า `settings.py`
- `pieces/`: จัดเก็บโครงสร้างชิ้นส่วนบล็อก Tetromino ได้แก่ `shapes.py`, คลาสออบเจ็กต์ข้อมูลโคตรรูปทรงเรขาคณิต `piece.py`, คลาสสร้างออบเจ็กต์ `factory.py`
- `systems/`: ระบบตัวช่วยทำงานเสริมในเกม เช่น หน้าที่จัดการหน้าจอคะแนน `score_manager.py`
- `utils/`: เครื่องมือใช้งานทั่วไป เช่น ระบบจานสี `colors.py`

# 7. OOP Design Explanation
การนำหลักการเขียนโปรแกรมเชิงวัตถุ (OOP) มาประยุกต์ใช้ออกแบบโครงสร้างเกม:

**1. Class**
มีการจัดกลุ่ม Object และพฤติกรรมในเกมอย่างเป็นหมวดหมู่ 
- `Game`: คลาสหลักสำหรับควบคุมลูปเกม (Game Loop), จับ Event คีย์บอร์ด และจัดการสถานะต่างๆ
- `Board`: จัดการตาราง Grid บนหน้าจอ, จัดเก็บสีบล็อก และคำนวณการล้างแถว (Line clear)
- `Piece`: ตัวแทนของชิ้นส่วนบล็อก จัดเก็บรูปทรง, สี, ตำแหน่ง และการคำนวณหมุน
- `ScoreManager`: คลาสแยกเพื่อจัดการระบบโชว์คะแนนและ Level ดึงตัวแปรออกจากเกมหลัก

**2. Inheritance**
ระบบสืบทอดคุณสมบัติ (Inheritance) เพื่อจัดการกับบล็อกแต่ละดีไซน์
- รูปทรง Tetromino ทั้ง 7 ชนิด (เช่น `IPiece`, `OPiece`, `TPiece`, ฯลฯ) ทำการ `extends` หรือสืบทอดคุณสมบัติมาจากคลาสแม่ `Piece`
- ทำให้ทุกรูปทรงมีพฤติกรรมพื้นฐาน (การขยับซ้าย/ขวา/ลง) เหมือนกัน แต่ต่างกันที่มีเพียงรูปร่างและสีเฉพาะตัว

**3. Factory Pattern**
การออกแบบระบบสุ่มแจกบล็อก
- ใช้คลาส `PieceFactory` ทำหน้าที่เป็นโรงงานผลิตชิ้นส่วน (Factory) 
- ตัวคลาสเดียวรับผิดชอบสร้างบล็อกแบบสุ่ม (Random) ส่งคืนกลับไปให้คลาสหลัก ทำให้คลาส `Game` ไม่ต้องไปปวดหัวกับกระบวนการประกอบร่างของ Block ชนิดอื่นๆ

**4. Single Responsibility Principle (SRP)**
โค้ดถูกออกแบบตามหลักการความรับผิดชอบเชิงเดี่ยว
- ทุกๆ Class ถูกมอบหมายหน้าที่เฉพาะตัวอย่างชัดเจน ไม่ทำงานเกินขอบเขต
- เช่น  `Board` ไม่สนใจว่าผู้เล่นกดอะไร (นั่นเป็นหน้าที่ของ `Game`) และ `Game` ก็ไม่สนใจว่าบล็อกหมุนยังไง (เป็นหน้าที่คำนวณของ `Piece`) ระบบจึงเป็นระเบียบและอ่านง่าย
